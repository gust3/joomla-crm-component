4. Архитектура/БД

- Опишите модель данных (таблицы/сущности/индексы).
```sql
-- Таблица компаний
CREATE TABLE IF NOT EXISTS `#__crm_companies` (
                                                  `id` int(11) UNSIGNED NOT NULL AUTO_INCREMENT,
    `name` varchar(255) NOT NULL DEFAULT '',
    `current_stage` varchar(50) NOT NULL DEFAULT 'Ice',
    `discovery_data` text DEFAULT NULL,
    `created` datetime NOT NULL,
    `modified` datetime NOT NULL,
    PRIMARY KEY (`id`)
    ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 DEFAULT COLLATE=utf8mb4_unicode_ci;

-- Таблица событий (история)
CREATE TABLE IF NOT EXISTS `#__crm_events` (
                                               `id` int(11) UNSIGNED NOT NULL AUTO_INCREMENT,
    `company_id` int(11) UNSIGNED NOT NULL,
    `event_code` varchar(50) NOT NULL,
    `comment` text,
    `created` datetime NOT NULL,
    INDEX `idx_company` (`company_id`),
    INDEX `idx_created` (`created`),
    PRIMARY KEY (`id`)
    ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 DEFAULT COLLATE=utf8mb4_unicode_ci;
```

- Поясните, как бы вы масштабировали решение до ~10k компаний/день (принципы: индексация, журнал событий, минимизация блокировок, подход к логированию).

Я не стал менять базу данных которую предложил ИИ она в принципе нормальная, но для сложных больших проектов - вероятно стоило бы в датах сделать timestam, так же статусы сделать int.
Вероятно стоило бы вынести архив в отдельные таблицы (может быть даже сервер если будут большие нагрузки - тогда можно было бы уменьшить таблицы и переносить события и компании на другой сервер)
Больше всего будет расти таблица событий - вероятно стоит ее партицировать.
Запись будет занимать тоже достаточно много времени - потому имеет смысл создать очередь записи. То есть при добавлении компании будут например добавлятся в Reids а оттуда будут уже сразу пачками загружаться - что снизит нагрузку на сервер. Можно будет заливать сразу по 200-300 компаний, что снизит нагрузку на сервер.
Для выборок и уведомлений вероятно стоит так же использовать Redis.
Если все совсем будет плохо можно подумать о шардировании.
